### How to deploy

Use [deploy.ps1](deploy.ps1) — laptop is source of truth for prod env via `.env.prod` (gitignored, template at [.env.prod.example](.env.prod.example)).

Common flows:
- `./deploy.ps1` — ship current version, push env, restart, verify
- `./deploy.ps1 -Patch` (or `-Minor`/`-Major`) — bump version via PR, then deploy
- `./deploy.ps1 -PushEnvOnly` — env change only, no image bump
- `./deploy.ps1 -PushEnvOnly -RotateSecrets` — regenerate ADMIN_TOKEN + SESSION_SECRET + KEK in `.env.prod`, push, restart
- `./deploy.ps1 -ImportEnv` — pull VM's current env down to `.env.prod` (bootstrap)

### Architecture conventions to honor

- Tz-naive UTC for all DateTime columns. Use `app._time.utcnow` everywhere.
- Service-layer is framework-free (no FastAPI imports). HTTP plumbing in `app/routers/*` only.
- All admin UI routes go through `require_login` + `require_csrf` for POSTs.

- Logger names: `license-server.<area>` (e.g. `license-server.boot`, `license-server.stripe`).

## v0.11.0 scope — production hardening

Three tiers. Tier 1 is the must-ship. Tier 2 is nice-to-have in the same PR if scope allows; otherwise spin off. Tier 3 is explicitly out of scope here — record but don't build.

### Tier 1 (must-ship)

#### T1-A: Automated DB backups to GCS

**Why:** The SQLite file is the entire crown jewels. If the VM disk dies, every license + private key is gone. ASM clients have the old pubkey baked into their image; we can't reissue under the same product slug because that requires the original private key. Mitigation today: zero. Risk class: data loss.

**Approach:** Nightly systemd timer on the VM runs a backup script that:
1. Uses `sqlite3 /var/lib/yg-license-server/license.db ".backup /tmp/license-snapshot.db"` to take a consistent snapshot while the server is live.
2. `gzip` the snapshot.
3. `gsutil cp` (or `gcloud storage cp`) to a GCS bucket with object versioning enabled (so we keep history without managing retention manually).
4. Lifecycle policy on the bucket: delete versions older than 30 days.

Bucket name suggestion: `gs://yglicenseserver-backups` (project `yglicenseserver`).

**Deliverables:**
- `deploy/gcp/backup.sh` — the actual script (idempotent, exits non-zero on failure, logs to syslog)
- `deploy/gcp/yg-license-backup.service` + `yg-license-backup.timer` — systemd unit + nightly timer
- `deploy/gcp/install.sh` — install the new units, create the GCS bucket if missing, set lifecycle policy
- `docs/deploy/gcp.md` — section on backups including restore-from-backup procedure
- Smoke test: tail journalctl after manual `systemctl start yg-license-backup.service`; confirm a fresh object lands in the bucket

**Authentication:** The VM has a default service account. Grant it `roles/storage.objectAdmin` on the bucket (NOT project-wide). One-time `gcloud projects add-iam-policy-binding` step in install.sh, gated on idempotent existence check.

**Edge case to handle:** `sqlite3` CLI is NOT installed on the VM (we discovered this when verifying the rewrap). Backup script must `apt-get install -y sqlite3` if missing, or use Python's stdlib sqlite3 via `python3 -c` (already on the VM). Pick whichever keeps the script simplest.

#### T1-B: Rate limiting on `/v1/check` and `/admin/login`

**Why:** A leaked license key today can be probed against `/v1/check` unboundedly. The admin login is brute-forceable on `ADMIN_TOKEN` — constant-time compare doesn't help if there's no rate limit.

**Approach:** Use `slowapi` (FastAPI-flavored wrapper over `limits`). In-memory storage is fine for single-instance; if/when we go multi-instance, swap to Redis. Default limits:
- `/v1/check`: 60/min per IP
- `/admin/login`: 10/min per IP

Use the same IP-derivation logic as `_client_ip_hash` in [app/routers/api.py:67](app/routers/api.py:67) — honor `X-Forwarded-For` only when peer is loopback (Caddy is on loopback in our deploy).

**Deliverables:**
- `app/rate_limit.py` — limiter setup + the IP-key function
- Decorators applied to the two endpoints
- Tests: hit the endpoint 11 times in a tight loop, assert 11th returns 429
- Add `slowapi` to `pyproject.toml` deps

**Don't:** apply to admin JSON API routes (already behind a strong bearer token; rate limit there is friction without security gain).

#### T1-C: KEK rotation script (`rewrap_secrets --migrate-from-prev`)

**Why:** Today, rotating `LICENSE_KEY_ENCRYPTION_KEY` would brick the deploy — encrypted rows become un-decryptable. `LICENSE_KEY_ENCRYPTION_KEY_PREV` is written to `.env.prod` by `deploy.ps1 -RotateSecrets` but nothing reads it.

**Approach:** Extend [app/scripts/rewrap_secrets.py](app/scripts/rewrap_secrets.py) with a `--migrate-from-prev` flag:
1. Read both `LICENSE_KEY_ENCRYPTION_KEY` (new) and `LICENSE_KEY_ENCRYPTION_KEY_PREV` (old).
2. For each encrypted row, decrypt with PREV, re-encrypt with NEW.
3. Plaintext rows pass through unchanged (still encrypted on first run under new KEK).
4. After successful run, the operator removes `LICENSE_KEY_ENCRYPTION_KEY_PREV` from `.env.prod` and pushes again.

**Implementation:** Add an alternate `_fernet_prev()` helper in [app/keystore.py](app/keystore.py). Don't expose `decrypt_secret_with(kek=...)` publicly — keep it internal to the script.

**Deliverables:**
- The flag + logic in `rewrap_secrets.py`
- Tests covering: KEK rotation roundtrip; missing PREV → error 2; PREV equals current → no-op
- A short blurb in `.env.prod.example` documenting `LICENSE_KEY_ENCRYPTION_KEY_PREV`

### Tier 2 (bundle if scope allows)

#### T2-A: `events` table retention

**Why:** `events` grows unbounded. The `heartbeat` type dominates (every `/v1/check` writes one). At 1000 installs/day, that's 365k rows/year per product.

**Approach:** A management command `python -m app.scripts.prune_events --older-than-days N` that deletes `heartbeat`-type events older than N days (default 90). Status-change, issue, delete, webhook events stay forever. Add a systemd timer alongside the backup one to run weekly.

**Don't:** add ON DELETE CASCADE at the DB layer — keep the surgical Python control (we may want to archive instead of hard-delete later).

#### T2-B: Structured JSON logs

**Why:** Current format is line-oriented. Aggregation tools (Loki, Datadog, etc.) want JSON.

**Approach:** Swap `logging.basicConfig` in [app/main.py](app/main.py) to use a JSON formatter (write our own — `python-json-logger` is fine but adds a dep we don't otherwise need). Behind an env var `LOG_FORMAT=json` (default still text for dev). Request-ID middleware already injects `request_id` onto records; in JSON mode, that becomes a top-level field.

#### T2-C: `/readyz` reports KEK status

**Why:** Currently `/readyz` only pings the DB. If KEK got wiped, request traffic would 500 on signing, but `/readyz` would happily say "ok".

**Approach:** Extend [app/main.py:103](app/main.py:103) `readyz()`:
1. Confirm KEK is set (or that no products exist with encrypted secrets).
2. Sample-decrypt one product's `private_key_pem` to prove the KEK matches.
3. Return `{status, db, kek, sample_decrypt}` shape. Caller can route traffic on this.

### Tier 3 (out of scope — record only)

- Postgres migration path: `DATABASE_URL=postgresql+psycopg://...` accepted by config, untested end-to-end.
- Webhook delivery retry queue: today is best-effort sync; receiver outages silently drop events.
- Stripe events via background worker: sync handlers couple HTTP latency to Stripe payload processing.
- License-expiry email warnings, CSV export, search/filter on admin tables.
- Multi-instance / HA — gated on Postgres migration.

## Decisions (locked 2026-05-17)

1. **Backup retention: 30 days.** GCS lifecycle policy deletes versioned objects older than 30d.
2. **Rate limits:** `/v1/check` 60/min per IP; `/admin/login` 10/min per IP. Both use the loopback-aware XFF derivation from `_client_ip_hash` in [app/routers/api.py:67](app/routers/api.py:67).
3. **T2 scope for v0.11.0:** Include T2-C (`/readyz` reports KEK status). Defer T2-A (events pruning) and T2-B (JSON logs) to v0.11.1 — they need design discussion / aren't urgent.
4. **v0.12 next-pick:** **Webhook retry queue** (Tier 3 item). Highest customer-impact of the three; pure backend, no DB migration. Postgres-path and Stripe-worker stay deferred until volume justifies.
