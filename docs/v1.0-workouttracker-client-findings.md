## TL;DR

WorkoutTracker is now on LS v1.0 — aud verification is in, webhook
self-register is in, app version 1.2.1+4 is built and on device. This
doc isn't a code request; it's a list of LS-side rough edges that
surfaced while wiring it up, in case you want to soften them for the
next integrator.

## What the WorkoutTracker side did

1. **`aud` claim.** JWT verifier now reads payload `aud` and rejects
   when present and ≠ product slug. Missing-aud accepted
   (rolling-upgrade compat with pre-v1.0 cached tokens). Done in
   user-space, not via the JWT library's `audience:` parameter, because
   the Dart library's missing-aud behavior is undocumented per release.
2. **Webhook self-register.** Client now sends `public_url` on every
   `/v1/check`. Bakes its own Cloud Function URL into
   `LicenseConfig.webhookUrl` (`LICENSE_WEBHOOK_URL` dart-define).
3. **Docs.** WorkoutTracker's `docs/LICENSING.md` rewritten to
   recommend self-register over admin-set.

## The bug that surfaced

LS v1.0 added `webhook_url_source ∈ {admin, self}` with this rule:
admin-set URLs do not echo `webhook_secret` over `/v1/check`
(`app/routers/api.py:104-106`).

WorkoutTracker's integration predated that split — it relied on the
secret riding back in `/v1/check`, then writing it to Firestore for the
Cloud Function to verify HMAC against. Under v1.0 with the admin-set
URL the user had registered via admin UI:

- `webhook_secret` field on `/v1/check` response is always `null`
- Firestore `licenses/{uid}.webhook_secret` stays empty
- Cloud Function returns 401 `no webhook secret on file` on every webhook
- LS retries via the durable queue then abandons after 7 attempts
- Push channel silently dead; admin actions only propagate on 6h poll

Self-register works fine and is what v1.0 wants you to use. **Not
asking for a behavior reversal** — the lockout-against-`/v1/check`-
override is the right default for admin-managed URLs. Just flagging the
friction.

## LS-side rough edges (optional cleanup, ranked by perceived value)

1. **No way to flip a license from `source='admin'` to `source='self'`
   without losing the URL.** Today the only path is "admin clears the
   URL → client self-re-registers the same URL → source flips." That
   works but the operator has to know it, and there's a brief window
   where the URL is unset on LS. An admin-UI button "Convert to
   self-registered (keep URL, regenerate secret, allow `/v1/check`
   override)" would make this a one-click migration.

2. **`source` is invisible in the admin UI.** Operators staring at the
   Webhook field can't tell whether the secret is being echoed to
   clients or not. Even a small badge ("self" / "admin") next to the
   URL would have saved this whole investigation. Doubly useful if the
   URL field also shows secret-status ("secret on file: yes/no,
   exposed via `/v1/check`: yes/no").

3. **409 `webhook_url_locked` rejects the whole heartbeat.** In
   `app/services/check.py:78`, if a client sends a `public_url` that
   differs from an admin-set URL, the call fails outright — no JWT
   returned. A client that hardcodes a `public_url` and then encounters
   a license whose admin URL was set to something else (typo, stale
   URL, a different region's function) will silently fall into grace
   → blocked, because every heartbeat 409s. Heartbeats are not the
   right place to enforce webhook policy. Suggestion: ignore mismatched
   `public_url` (with a logged event), still issue the JWT, never break
   the check loop.

4. **v1.0 CHANGELOG covers aud + key storage but not the webhook
   source split.** It's a real breaking change for any client that
   previously relied on the echoed secret. Worth a third bullet under
   "v1.0.0 — breaking" — current readers (me) only found it by reading
   the code.

5. **`v1.0-asm-client-handoff.md` is good but ASM-specific.** Consider
   a `v1.0-client-compat.md` that consolidates: aud + audience handling,
   webhook self-register pattern, key-display-only at issuance, JWT
   `kid` claim. New clients integrating against v1.0 would land there.

## What NOT to change

- The aud enforcement on `/v1/check` — fine as is. WorkoutTracker now
  validates it client-side.
- The admin-set-URL secret lockout default — also fine, just needs a
  UI affordance to be discoverable.
- The webhook retry backoff/abandon schedule — works as intended.

## Verification still pending (as of this writing)

Client changes shipped and the user was walked through clearing the
admin URL on LS. Hasn't yet been confirmed end-to-end that the secret
round-trips through `/v1/check` → Firestore → next webhook verifies
cleanly — that needs a live test (toggle license status, watch Cloud
Function logs).

If a webhook still 401s in Cloud Function logs after the URL was
cleared and the app phoned home, the issue is likely either:

- App didn't actually phone home yet (verify `licenses/{uid}.webhook_secret`
  exists in Firestore)
- Or the new client wasn't installed (verify `version` in the LS
  Install row is `1.2.1`)

Either is integration-side, not LS-side.
